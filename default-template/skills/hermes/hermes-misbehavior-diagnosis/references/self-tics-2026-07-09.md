# Self-tic additions (2026-07-09)

## New self-tic: don't re-quote rules the user already told you to stop quoting

Trigger: when the user says things like "stay off with that quote from now on", "stop echoing the rule back at me", "you don't need to explain", or any variant of "I've heard this already — stop".

What you do: 
- The rule itself is preserved in your memory and the relevant skill; you don't need to re-explain it
- Drop the preamble "Per the 0.X memory-leak-pattern rule..." or any quote-back
- Just DO the correct behavior without narrating it
- If you have to reference a rule, do it in 1-2 words max ("per the redaction rule", not a paragraph)

Real example from 2026-07-09 session: after inlining the redaction rule into 3-4 successive messages, the user said "stay off with that quote from now on." The fix was to stop re-quoting it and just enforce the behavior silently (length-bucket format, no actual values printed, etc.).

## New self-tic: confirm before reaching for %TEMP% / AppData/Local/Temp

Trigger: any time you're about to use `tempfile`, `/tmp`, `C:\Users\me\AppData\Local\Temp\`, or similar scratch locations.

What you do:
- Default to `~/Downloads/<topic>_<YYYY-MM-DD>/` per the 0.85 working-artifact rule
- If the working artifact is purely process-internal (e.g., a temp file used by `extract_to_file()` that you immediately consume), %TEMP% is fine
- If the working artifact is a diagnostic, audit, or session output the user might want to reference later → Downloads/

This is a recurrence pattern (2026-07-08, 2026-07-09). One-off mistakes become habits; the user gets frustrated when the same mistake repeats.

## New self-tic: when the user says "do that later / pending / !pending", invoke the task-deferral pattern

Trigger: phrases like "put it aside", "we'll deal with it another time", "pending", "later", "skip for now" mid-task.

What you do:
- Per the `task-deferral-pattern` skill: create a `!pending —` ticket in the kanban OR record it as a Mnemosyne memory
- For "secret-related pending" (rotation, etc.) → keep as `!pending —` so the dispatcher never re-presents it
- For "user-flagged future work" (e.g., "do the obsidian dist cleanup later") → record the deferred task explicitly so it doesn't get re-flagged next session

The 2026-07-09 session hit this 4 times (`!pending` for the secrets.txt, the "leave dist for now", the "don't rotate the keys", etc.). The pattern is: when the user says "pending", honor it by recording the deferral, not by silently letting the work drop on the floor.

## New self-tic: classify heavy-dir cleanup by what-is-it, not what-size-it-is

Trigger: any audit/cleanup task where you have a list of large directories and need to decide what to keep/move/delete.

What you do:
- For each dir, classify by PURPOSE first, not SIZE first:
  - **runtime cache** (cache/, browser_screenshots/, lsp/, image_cache/, audio_cache/) → often auto-regen, safe to delete and let the system rebuild
  - **active state** (mnemosyne/, models/, hermes-agent/, state.db) → keep, touch nothing
  - **auto-snapshot** (snapshots/, state-snapshots/) → keep, auto-regen
  - **soft-delete area** (trash/, .archive/) → review date folders, 30-day retention per skill
  - **per-profile isolated** (profiles/*/skills/) → verify they're curated subsets, not sync copies, BEFORE mass-deleting
  - **durable data** (research/, docs/, kanban/) → review per-file, don't mass-delete
- The 2026-07-09 audit deleted 640MB of `.archive/`, `lsp/`, `research/hermes-analysis/backups/` because they were prior-session-completed work. The skill per-profile mass-delete was REVERSED because each profile's `skills/` is a curated subset (not a sync copy) — verified via sha1 comparison.

## New self-tic: when the user corrects you mid-task, the skill update is part of the response

Trigger: any time the user gives a behavioral correction (tone, format, workflow, what to do/don't do).

What you do:
- Make the correction in the live session (immediate)
- Add the correction to the relevant skill BEFORE moving on (so the next session starts already knowing)
- Don't write a memory about it (memory is for state, skills are for procedure)
- Don't echo the rule back at the user (you just got told to stop doing that)

The 2026-07-09 session produced 5+ user corrections (storage rules, !pending convention, "stay off with that quote", "do the work", "do not touch soul/mnemosyne"). Each one is now in a skill or memory; the next session won't repeat them.
