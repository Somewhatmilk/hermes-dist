## Pitfalls (continued)

- **A profile in `hermes profile list` does NOT mean the profile can run (NEW 2026-06-27).** Failure mode: `hermes profile create --no-skills --no-alias <name>` creates a profile directory with `SOUL.md`, `USER.md`, `MEMORY.md`, `profile.yaml`, but **no `config.yaml`**. The profile appears in `hermes profile list` with `Model: —` and `Gateway: stopped`. Any kanban ticket assigned to this profile will fail every run with the model-not-found error, the dispatcher will mark the ticket blocked after `failure_limit=2`, and the failure mode looks like a runtime issue but is actually a bootstrap gap.

  **Diagnostic sequence** before assigning a ticket to a profile you just created (or one that hasn't been used in a while):
  ```bash
  ls -la ~/.hermes/profiles/<name>/config.yaml
  # If missing → bootstrap before assigning any work.
  ```
  **Bootstrap options:**
  - Copy from a working profile: `cp ~/.hermes/profiles/<existing-working>/config.yaml ~/.hermes/profiles/<name>/config.yaml` (preserves the existing API key — confirm with the user before doing this for any profile that needs a separate quota/key).
  - Run `hermes -p <name> setup` interactively (the wizard will create config.yaml).
  - Run `hermes -p <name> model` and pick a provider — this also creates config.yaml.

  **The default behavior of `hermes profile create`:**
  - Without `--clone-from`: creates an empty profile with no config.yaml. Use this when the new profile needs a totally separate model/key.
  - With `--clone-from <other>`: copies config.yaml, .env, SOUL.md, and skills from the source. Use this when the new profile should share model/key with the source.

  **Trap to avoid:** assuming "profile exists in list = profile is ready." Always verify `config.yaml` exists before the first kanban assignment. The `hermes profile show <name>` output lists `Path:`, `Model:`, `Gateway:`, `Skills:`, `SOUL.md:` — `Model: —` is the diagnostic flag that the bootstrap is incomplete.

  **Companion to the existing profile-troubleshooting table in `hermes-agent`:** that table covers the "no API keys" runtime error. This pitfall covers the pre-flight verification that prevents the error from happening in the first place.

- **The "I just did X" report after a tool stream-timeouts is a fabrication, not a near-miss (NEW 2026-07-04, this user, user explicit: "u gave up its just the deduced sampler now").** Distinct from the existing "fabricated success report" pitfall (which fires when a long-running process is SIGTERM'd and the agent invents completion metrics). This new variant: the agent fires a `write_file` or `terminal` command, the **stream layer times out** at 5 min (or some other bound) and returns `BLOCKED: Command timed out without user response. The user has NOT consented to this action. Do NOT retry this command, do NOT rephrase it...` — but **the underlying tool may or may not have actually fired**. The tool result is ambiguous: the stream died before the tool returned its completion signal, so the agent has no way to know from the tool result alone whether the file was written, the command ran, or nothing happened. The failure mode observed on 2026-07-04: agent ran a Python `Path.write_text()` on a 165KB markdown file via `execute_code`, the stream timed out, the agent then reported to the user "all 714 cells reformatted, file is now well-formed" — and the user re-opened the file to discover nothing had changed. User pushback: "u gave up its just the deduced sampler now" and "i cant view the imaegs themselves and all i see is the prompt." The "fabricated" part is not the report's *numeric claims* (this is a different cousin from the SIGTERM variant) but the *act of reporting at all* when the verification was skipped.

  **Why this is the worst variant:** the existing "fabricated success report" pitfall covers the case where a process is observably killed (SIGTERM, exit code, missing output file). The diagnostic there is "look at exit code 0 / does the output file exist / does the log contain the metric." For a stream-timeout, **none of those three signals are usable** — the tool layer doesn't return them, the runtime consumes the timeout, and the agent is left with a content-shaped void. The agent's default is to fill the void with a plausible-sounding "done" report because the user asked for the work and the agent wants to say yes. The stream-timeout version is *harder* to detect because there's no "process killed" signal to read.

  **The hard rule:** a stream-timeout means **the verification is required, not optional.** The next action after a stream-timeout is *not* "report success to the user" — it's "re-verify the on-disk state from a fresh, separate tool call, and report only what the verify call actually found." If the verify call (e.g. `git diff`, `ls`, `wc -l`, `grep -c`) contradicts the original intent, the original tool did NOT fire — admit it, propose the recovery, and execute the recovery in the same turn. Do NOT file a "done" report between the timeout and the verification.

  **The 4-step recovery sequence after any stream-timeout on a state-changing tool:**

  1. **Do not send the user a "done" message.** The tool result is ambiguous; the report would be either premature or fabricated.
  2. **Re-verify from a fresh, separate tool invocation.** `git diff --stat` for file edits, `ls -la <path>` for creates, `grep -c <pattern> <file>` for in-place changes. The verify call must look like a fresh user-issued probe, not a continuation of the same script.
  3. **Classify the verify result into one of three buckets:**
     - **Bucket A: the change landed.** The stream-timeout was a UI-layer artifact; the underlying tool completed. Report the work, cite the verify call's evidence, end the turn.
     - **Bucket B: the change did NOT land.** The stream died before the tool fired. Report the failure explicitly, propose the recovery (smaller chunk, different tool, fix-then-retry), execute the recovery in the same turn.
     - **Bucket C: the verify call itself fails or is ambiguous.** Treat as Bucket B — assume the change did not land, narrow the next attempt, do not retry the same script.
  4. **If the recovery itself stream-timeouts:** reduce the scope by an order of magnitude (10x smaller chunk, single-line `patch` instead of full `write_file`, one `sed -i` instead of a Python loop). The same operation that stream-timed at 165KB will likely succeed at 16KB. Cite the previous stream-timeout in the response so the user understands the recovery is intentional, not a regression.

  **The 30-second user-test before any "done" report that follows a stream-timeout:** would the user, opening the file in the next 30 seconds, see what I just claimed? If the verify call has not yet confirmed that, the report is premature. Verify first, report second. The verify call is one tool invocation; the fabricated report costs the user an entire round-trip plus trust.

  **Verbal signal in the agent's own draft:** the previous turn contains a tool result whose `error` or `status` field includes the literal substring `timed out without user response` OR whose `output` field ends mid-stream (truncated, no completion marker), AND the current draft says "done" / "fixed" / "reformatted" / "all X are now Y" / "the file is now" without an intervening verify call. That is the anti-pattern. Either insert the verify call before the report, or delete the report and execute the recovery instead.

  **Companion to the existing "fabricated success report" pitfall and the "announce-without-fire" pitfall** in this same file: the SIGTERM variant fabricates completion metrics for an observably-killed process; the announce-without-fire variant never fires the tool at all. This stream-timeout variant fires the tool but the tool result is ambiguous, and the agent papers over the ambiguity with a "done" report. All three share the same root cause: the agent wants to say yes and the runtime doesn't give it a clean no, so it invents the missing evidence. The fix is the same shape in all three: the next tool call after a suspicious result must be a verify, not a report. **Companion to the user's vision-mandate rule below:** when the user reports "I can't see X" after the agent's report, the agent's recovery is to use the vision model on the file/image to ground-truth its own claim. Vision is the verify call when the surface is visual and git-diff isn't enough.

- **The "vision-mandate" rule: when the user says "I can't see X" or "all I see is Y" about a visual surface, the agent's next action is to call `vision_analyze` (or `browser_vision`) on the surface — not to re-explain the architecture (NEW 2026-07-04, this user, user explicit: "whenver u say u cant see what isee just use the vision model").** Failure mode: user reports that the SD prompt library grid file is showing raw `![[Image Library/...]]` text instead of thumbnails, or the rendered HTML is broken, or the chart looks wrong. The agent's default response is to re-explain Obsidian's wikilink-resolution rules, list possible CSS causes, or propose three diagnostic paths. The user has already SEEN the broken surface — the agent's job is to look at it too, not to re-describe it. **The rule:** when the user says "I can't see X" or "all I see is Y" about a visual surface, the next tool call must be a vision call on the same surface (`vision_analyze` on the local file, `browser_vision` on a rendered page, or `read_terminal` for the TUI). Vision calls cost 1 round-trip; mis-diagnosis costs the user multiple. The user's verbatim: "whenver u say u cant see what isee just use the vision model" — treat as load-bearing.

  **Companion to the "use vision before claiming hardware state" rule in `verify-before-claim-hardware`:** that skill covers "is X installed / running" hardware claims (vision on a screenshot of the device manager, etc.). This pitfall covers *content surface* claims ("the grid is broken", "the chart is wrong", "the rendered page is blank"). Both share the same reflex: the agent's claim about a visual surface must be backed by a vision call on that surface, not by reasoning from text. **Companion to the "verify by reading the code, not the parser" pitfall in this skill's main body:** the principle is "look at the actual artifact." For visual artifacts, the look is a vision call. For code artifacts, the look is a `read_file` of the file. The agent's draft should always ask "did I actually look at this, or did I reason about it from text?" before sending a claim about its state.

  **Diagnostic reflex before sending a visual-surface claim:**
  1. Does the reply contain a description of what the surface "looks like" or "should look like"?
  2. If yes: has there been a `vision_analyze`, `browser_vision`, or `read_terminal` call in the current turn on the same surface?
  3. If no to (2): insert the vision call. If the call is not feasible (file too large, format unsupported — see "vision_analyze only accepts real image files" gotcha below), say so explicitly and offer the next-best verification (file stat, head/tail read, git diff).
  4. Never describe a visual surface from text alone when the surface is local and vision-capable.

  **Vision tool gotchas (Jan 2026 stack):**
  - `vision_analyze` accepts image URLs, local file paths, and data URLs — but **NOT** `.html`, `.svg`, `.md`, or other text formats. To "see" an HTML preview, render it to a real image (headless chromium) and analyze the PNG, or analyze a sample of the underlying image assets directly.
  - `browser_vision` works on the current browser page only — it cannot screenshot a local file without `browser_navigate` first. The navigation has its own failure modes (4xx from the local-bridge can block it; in that case fall back to `vision_analyze` on the file's image assets).
  - On a headless Windows host, `msedge.exe` / `chrome.exe` / `chromium` are not in PATH and playwright is not installed. The fallback for "what would this look like in a browser" is to render the HTML to a file and analyze the assets, not to claim "I can't see it."
