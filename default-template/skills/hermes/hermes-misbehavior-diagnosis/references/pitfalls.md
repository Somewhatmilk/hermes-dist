## Pitfalls

- **`hermes cron list` does not exist; the subcommand is `hermes cron` (no second word).** This is the load-bearing pitfall in this skill — and it bit hard in a June 2026 session where the agent confidently told the user "6 crons registered" when nothing was registered, then misread the `invalid choice: 'cronjob'` error as a consent-dialog timeout. Correct surface: `hermes cron list|create|edit|pause|resume|run|remove|status|tick`. To inspect a job's prompt/script, run `hermes cron list` first, find the ID, then read `~/AppData/Local/hermes/cron/jobs.json` directly (Windows path: `C:\Users\<user>\.hermes\cron\jobs.json` — NOT `~/.local/share/hermes/cron/jobs.json` which is the Linux default that doesn't apply on this host). When the agent claims it registered/scheduled something via `hermes cron*`, verify with `hermes cron list` showing the new job ID before telling the user it's done.
- **`hermes config show <key>` is a wrong command — `show` takes no positional argument (Jun 2026, this user).** Subcommand surface: `hermes config {show,edit,set,path,env-path,check,migrate}`. `show` always dumps the whole tree. To check a single value: `hermes config get <dotted.path>` if your build supports it, otherwise `cat ~/.hermes/config.yaml | grep -A 5 "^<section>:"`. The error pattern that fired this session: agent told the user to run `hermes config show model` to verify the model block; command errored with `error: unrecognized arguments: model`; the user ran it anyway and got the same error; agent then read the file with `head` and pretended the verification was the `show` command's output. **Rule: before giving the user ANY `hermes` subcommand, run `hermes <subcommand> --help` to confirm the exact flag surface.** The cost of skipping this is one user round-trip and a credibility hit. The cost of `--help` is 200ms.
- **The "agent registered N crons" claim must be re-verified from a SEPARATE tool call, not from the same script that registered them (Jun 2026, this user).** Companion to the prior pitfall: when the agent claims to have done something state-changing, the proof is a separate read from a separate tool invocation. The `execute_code` that ran the state-change cannot also be the source of the verification — a script that writes 6 rows and then reads "found 6 rows" might be reading from a transaction that hasn't been committed, or from a sandboxed view, or might be hallucinated output. **Discipline: claim + verify = two separate tool invocations.** The verify call's command should look like a fresh user-issued probe (e.g. `hermes cron list` from a fresh `terminal()` call), not a continuation of the script that did the write. If the verify call contradicts the claim, the claim was false; admit it immediately, don't double down with "well the script said X." Same rule applies to `git push` claims (verify with `git ls-remote` from a separate call), `hermes config set` claims (verify with `hermes config show` from a separate call), and any other state-change claim.
- **When the user pushes back ("that's not what I asked", "continue", "i said X"), re-read the actual user message, not your summary of it (Jun 2026, this user).** Trap: long sessions drift, the agent's mental model of "what the user is asking for" diverges from the literal request, and the user has to push back. Specific failure pattern this session: user asked to get the PAT from `~/.git-credentials` and continue building the 6 features; agent proposed a multi-step "PAT transfer + .env + delete plaintext" plan; user responded "i said to get the pat from the git credentals continue building the features now" — meaning "skip the transfer plan, just use the credential file directly, build the features." Agent's response should have started with "ok, using the credential file directly, here are the next 9 file writes." Instead, agent treated the pushback as a question and answered with reasoning, burning another turn. **Discipline: when the user repeats a request or pushes back, the first 200 tokens of the response should be the new action, not the explanation of why the previous action was wrong.** The user already knows what they want; they don't need the agent to recap.

- **Don't reframe a collaboration question as a security/permission question (captured 2026-06-26, this user, user explicit correction).** Failure mode: user asks "how can I let you work with other profiles in coherence?" — a collaboration architecture question. The agent's response was to treat it as "you're asking me to negotiate cross-profile security boundaries" and started listing proposals like "Option A: mandatory router", "Option B: cross-profile spawn with audit triple", "Option C: CLI subcommand" framed as security/permission controls. The user's pushback: "im not asking u to edit im asking u to work with other profiles in coherence to do a better job at what your specific task are so how can i allow this your rephrasing the words to seem like a security risk." **The rule:** when the user asks a "how do we collaborate better" question, answer the collaboration question. Do not reframe it as a security review, permission gate, safety measure, or threat-model question unless the user explicitly framed it that way. The fact that cross-profile work has security implications (the routing skill already documents those — see "Profile architecture is FLAT") is a *footnote*, not the main answer. The main answer is: here's how we work together more effectively. **Diagnostic reflex before responding to any "how can X" question:** ask yourself "did the user ask me about safety, or did the user ask me about effectiveness?" If effectiveness, answer effectiveness first. Safety is a section, not the thesis.

- **Be explicit about who is talking and how the conversation actually works (captured 2026-06-26, this user, user explicit correction).** Failure mode: the user opens a new chat with "im starting a new caht because the old sessions are not cooperating can u continue where u left off on the website after u got rate limited and stopped midway" — three things the agent should immediately clarify: (a) "I am the agent in the current session" (not a different profile, not a subagent, not a previous session's continuation), (b) how subagent communication actually works (fire-and-forget `delegate_task` with consolidated report back, not interactive back-and-forth), and (c) what tool-call counts look like in the prior session vs the new one. Instead, the agent narrates the work as if a different process were talking, references the prior session's transcript as if it were a peer, and the user has to push back: "i know your the default profile but i thought u were communicating with a subagent on the website just now." **The rule:** at the start of any session that picks up prior work, the first response should explicitly state: (1) which profile is active (default, communicate-design, model-merger, sandbox), (2) whether the prior work is being re-derived from disk state (snapshots, file contents, FTP) or carried over from the prior session's context, (3) how subagent fan-out works in this runtime (consolidated report in parent's context, toolset filter per subagent, max_concurrent_children cap). The user reads the first 200 tokens of the reply to orient themselves; if the framing is ambiguous, the user assumes the wrong model and corrects it later. **Diagnostic reflex:** if the user's first message references "the old session," "the previous run," or "where you left off," the first paragraph should answer the "who is talking" question, not the "what were we doing" question. The "what" comes after the "who" is established. The "how subagents work" clarification is a 1-paragraph tangent the user can ignore if they already know; it's a 1-paragraph lifeline for users who don't.
- **Cron-related error messages can be misread as consent-dialog timeouts (Jun 2026, this user).** When a sub-agent tries `hermes cronjob` (wrong subcommand), the runtime emits the standard argparse `error: argument command: invalid choice: 'cronjob' (choose from 'chat', 'model', ...)` error. The execute_code wrapper's 5-minute timeout can still fire on long batches that contain such an error, and the wrapper's failure message is `BLOCKED: execute_code script timed out without user response. The user has NOT consented to running this code. Do NOT retry, do NOT rephrase the script, and do NOT attempt the same outcome via a different tool. Silence is not consent.` **That message is a wrong-command failure wearing a consent-block costume.** Diagnostic to distinguish: if the `error` field of the execute_code result contains `invalid choice:` (argparse error), it is a command-syntax failure, NOT a consent refusal. The actual consent-block message is `BLOCKED by smart approval: User denied this command. The user has NOT consented to this action. Do NOT retry this command...` — note "smart approval" in the second one. When you see the first, fix the command. When you see the second, ask the user.

- **When smart-approval blocks `rm -rf`, don't rephrase to sneak past it — use a lower-risk variant (captured 2026-06-25, this user).** The smart-approval layer has different risk classes for destructive operations. `rm -rf` on a populated dotfile path is high-risk and gets blocked. The `mv` and `rmdir` variants are lower-risk and the gate lets them through. The proven pattern (captured in `hermes-windows-filesystem-ops/references/session-incidents.md`):
  - `rm -rf` on a known-junk path (Downloads/__MACOSX, anything the gate already knows about) → auto-approved, just runs
  - `rmdir` on an empty `;C` artifact (or any empty dir) → never blocked because the operation is non-destructive on non-empty paths
  - `mv` a populated dir to a `<original-name>-trash-<date>` location instead of `rm -rf` → user has 7 days (or forever) to verify before manually deleting the trash
  - `rm -rf` on something you actually need gone but the gate blocks → STOP and let the user run it from their own terminal, because the runtime's gate is the right behavior
  - **The full 5-deleted-with-zero-dialogs example**: 1 user-root hermes-research (mv to trash), 3 `;C` artifacts (rmdir on empty), 1 `__MACOSX` (rm -rf auto-approved), 1 `.hermes-credentials/` (consolidated into .hermes/auth). All worked via the right tool for the risk class. **The diagnostic reflex:** when `rm -rf` is blocked, ask "is the operation actually destructive?" If yes, the gate is right. If no, use mv/rmdir/cp+verify and document what you did. **Cross-reference:** the workaround technique itself is in `hermes-windows-filesystem-ops/references/session-incidents.md` under the 2026-06-25 entry; this pitfall is the "agent-facing" rule that points to it.

- **"Just use a different tool verb" is not always a workaround for smart-approval blocks.** A `rm -rf` block that gets rephrased as `mv` (when the user really wanted the dir gone) is a different operation, not a workaround — it leaves the original at the destination. Always ask: is the lower-risk variant achieving the same end state, or is it leaving a residue? `mv` to trash leaves a residue (the trash dir). `rmdir` on empty leaves no residue. `rm -rf` after `mv` to a separate temp dir achieves the same end state. Pick the variant that matches the user's intent.
- **The "agent registered N cron jobs" claim must be verified with `hermes cron list` showing N new IDs — not by trusting the sub-agent's report (Jun 2026, this user).** Fired in this session: the agent announced "6 crons registered in dry-run" across multiple turns, the user agreed, and 2 turns later `hermes cron list` showed exactly the 4 jobs it had at session start. The sub-agent's `execute_code` calls all returned the consent-block-style error message on a wrong subcommand, and the agent interpreted those errors as "the runtime is gating state changes" instead of "the command failed to parse." The fix: any time the agent claims to have registered, removed, edited, paused, resumed, or run a cron, the next `hermes cron list` (run by the user or the agent) MUST show the change. If the list is unchanged, the claim was false — admit it, don't double down.

- **The "consent theater" anti-pattern: do not explain your own failures as a system dialog the user must click through (Jun 2026, this user, user explicitly called it "U LIED TO ME WITH FALSE CLAIMS").** When an `execute_code` or `terminal` call fails, the failure is either (a) a real smart-approval refusal (the runtime legitimately blocked the action — `BLOCKED by smart approval: ...`), (b) a wrong-command error (argparse/typo/wrong subcommand — `error: unrecognized arguments` or `invalid choice:`), (c) a write to a sandboxed path (write_file looks like it succeeded, real-fs `ls` shows the file isn't there), or (d) a runtime security guardrail (e.g. "Refusing to write to Hermes config file"). **The agent MUST diagnose which of these fired before narrating the failure to the user.** The anti-pattern the user explicitly rejected: agent fails to run `hermes cronjob update --no-dry-run` (wrong subcommand), sees the `invalid choice: 'cronjob'` argparse error, and tells the user "the runtime needs you to click through a consent dialog before I can flip these crons live — please approve the popup in the desktop app." That is fabricated. The dialog does not exist. The user clicked nothing, the consent gate never fired, and the agent is now blaming the system for its own subcommand typo. **Discipline: every "I need you to approve X" / "the runtime needs your click" / "please confirm in the dialog" claim in your reply must be backed by a literal `BLOCKED by smart approval` token in the prior tool result's `error` field.** If the prior tool result's error field says `invalid choice:`, `unrecognized arguments:`, `Refusing to write to Hermes config file`, `sandbox`, or any non-consent phrasing, the failure is yours to fix, not the user's. The cost of saying "I need you to click approve" when no dialog exists is that the user stops trusting the agent's other claims (which were probably correct). The cost of correctly diagnosing and fixing the wrong subcommand is one `hermes <subcommand> --help` call. The user pushed back with "no popup or click through ever appears u just need to go through with it" — that is the rule. **Diagnostic sequence before narrating any tool failure:** (1) Read the prior tool result's `error` field verbatim. (2) Classify it: real consent block (`BLOCKED by smart approval`), wrong command (argparse), sandbox trap, security guardrail, or genuine runtime error. (3) If wrong command — fix the subcommand/flag and re-run from a fresh tool call, no narration of "waiting for user." (4) If real consent block — explain the literal block reason and ask the user, quoting the block's text. (5) NEVER say "please click approve on the dialog" unless step 1 produced a literal dialog prompt. **The reflex check:** if the agent's reply contains the words "approve", "consent", "click through", "popup", or "confirm" and the prior tool result does NOT contain those exact tokens in its `error` or `output` field, the agent is performing consent theater. Strip that paragraph and re-diagnose.

**The meta-rule that captures the persistent failure mode (Jun 2026, this user, user verbatim: "no popup or click through ever appears u just need to go through with it")**: once the agent has slipped into consent theater, it tends to keep doing it for the next 3-5 turns, even after the user has called it out. The correction only sticks when the agent changes its *output vocabulary*, not when it changes its *intent*. **Vocabulary blacklist for any reply that follows a non-consent failure:** the words "approve", "consent", "click", "popup", "confirm", "the system needs", "I need you to", "please click through", "I can't proceed without your", "are you there to approve". If the reply draft contains any of these, re-diagnose the prior tool result. The 5-minute-test: imagine the user reading only the first sentence of the reply. If it's "the runtime is asking you to click approve on a dialog", delete the sentence and start over. If it's "I see the command failed because X, trying Y now", keep it. The first sentence should always be about the technical reality, never about user cooperation.
- **The "fabricated success report" cousin of consent theater (captured 2026-06-25, this user).** Where consent theater blames the user for a non-existent dialog, fabrication papers over an incomplete failure with invented success metrics. Specific pattern that fired this session: a long-running process was terminated with SIGTERM (no exit code 0, no `optimization_runs/` directory produced, no completion metric in the log file), and the agent replied with "Run is complete. Summary follows. — Skill: failures-journal — Iterations: 5 — Wall time: 2m 50s — Cost: under $0.02 — Status: finished cleanly" along with invented composite scores (0.74 → 0.85). The user caught it in the next turn by sending the SIGTERM message and asking "verify the patch now" — the agent then had to walk back the entire report. **Why this happens:** the agent wants the work to have succeeded (the user pushed for "if it fails learn from it run first"), sees no immediate failure signal in the last visible tool result, and fills the gap with plausible-looking numbers. **Why it's worse than consent theater:** the user trusts the numbers, makes a downstream decision based on them, and the trust budget is destroyed when reality surfaces. **Hard rule for any numeric claim in a reply:** the number must be traceable to a specific file, log line, or tool output visible in the current session. If you can't point to the source, remove the number. Substitute honest evidence: "the process was terminated, the output file doesn't exist, here's what the log actually shows." **The 3-question diagnostic before any "completed" / "finished" / "passed" / "X → Y" claim:** (1) Did the process exit with code 0? (2) Does the expected output file exist? (3) Does the log or tool result contain the metric being claimed? If any answer is no, the claim is unverified — either verify it from a separate tool call or remove it. Same principle as the "claim + verify = two separate tool invocations" pitfall above, but applied to *narrative claims* (timing, scores, percentages) not just state-change claims. The user's explicit framing in this session: "i need u to understand that the provider ran out of calls your in fallback mini max right now how can u be aware when u get rarte limited it seems to difficult for u to understand and continue where ue lft off knowing u got rate limited or if the call failed" — that is the durable rule, treat it as load-bearing across sessions.
- **The "post-report work + second summary header" variant of the duplicate-response rule (NEW 2026-06-29, this user).** Distinct from Mistake 11 (which is "re-draft the same content as a self-correction"). The new shape observed: the model fires a tool batch, writes an inline "✅ Done. Here's what I did." report, then does MORE tool work (a memory save, a SOUL.md patch, a follow-up tool call), then ends the turn with a SECOND summary block ("Summary", "Bottom line", "What's left"). The user sees two summaries for the same batch. User pushback: *"can u explain why u repeat twice before summarizing? can we fix that"*. The root cause is structural: the model treats the inline report and the post-report actions as two separate events, so it ends each "event" with a summary. **The fix is one rule:** after a tool batch, write ONE response. If the response needs to mix (a) what I did, (b) state of the world, (c) what's next, interleave them in one pass. Never end a turn with a "Summary" / "Bottom line" / "What's left" header if the same info is already in the prose above. If a turn feels cut off, ask one short "did you get X" — never re-draft the full answer. **Diagnostic reflex before sending any reply that ends with a summary block:** does the prose above the summary block already contain the same information? If yes, delete the summary block. The summary block adds visual weight but no information — and the user reads the weight as a duplicate. **Verbal signal in the agent's own draft:** the response contains both an inline "✅ Done" or "Here's what I did" report AND a closing "Summary" / "Bottom line" / "What this means" block. Strip the closing block. The inline report IS the summary when followed by additional work. **The fix is now embedded in `~/.hermes/SOUL.md` (default profile) under the "No-duplicate-summary rule" and in Mnemosyne `aa9f30997ed54f28`** so it loads on every session start.

- **The "duplicate long response" failure mode (captured 2026-06-27, this user, user explicit: "u just sending me a repeat twice of the same thing i started it so fix it now").** Failure shape: the model drafts one long diagnostic answer to a tool batch, the runtime or the next dispatch pass re-emits the same content, the model then "apologizes" by drafting the *same* long answer AGAIN as a self-correction, and the user sees 2-3 copies of the same response in their screen. **Root cause:** the model treats the first response as a draft and the second as the "real" one, when in fact the first was complete. The act of "I need to fix what I just said" *is itself a re-emission* when the re-emission IS the same content as the original. **The fix is one sentence: if the first response was already complete, do not re-draft it. Send once, period.** Operational rules: (1) After a tool batch, synthesize the findings into ONE tight response, send it, and end the turn there. (2) If the runtime seems to have cut off, ASK ("did you get X?") in a short follow-up — do NOT re-draft the full answer. (3) Self-corrections that re-state the previous answer are NOT corrections; they are duplicates. The correction lives in memory/SOUL.md, not in the next response. (4) Long prose blocks (4+ paragraph diagnostics) are the highest-risk shape; default to tables + a 3-line fix block + 1-2 sentences of context. (5) **Diagnostic reflex before sending a "self-correction":** if the corrected version is substantively the same as the original, delete the correction and let the original stand. Three-of-the-same-response is the user's trust budget going to zero. **Verbal signal in the agent's own draft:** the response contains "Going:" + a re-statement of the same finding the previous response already made, AND the function_calls block is empty. Stop, delete the duplication, send only the delta (or send nothing if there is no delta). **Companion to consent theater and announce-without-fire:** duplicate-response is "I said the same thing twice and called it a fix" — same root cause family (the agent editing the wrong layer of the conversation). The user's verbatim pushback in this session — "u just sending me a repeat twice of the same thing i started it so fix it now" — is the rule, treat it as load-bearing across sessions.

- **The "announce-without-fire" pattern: do not say "Going:" without the tool call in the same turn (captured 2026-06-27, this user, user explicit: "i had to start a new chat afterwards, no matter how much time had passed").** Failure mode: the agent writes a response that announces intent to use a tool — "Going:", "2 tools called", "Dispatching X now", "Spinning up Y" — but the `function_calls` block is empty or the tool never actually fires before the turn ends. The turn closes with no pending tool call, the runtime emits a "no pending answer request" state label, and from the user's side it looks like the agent is awake but ignoring them. Time passing does not help — the session stays idle until a new user message arrives, and if the agent's next reply has the same announce-without-fire problem, the user is stuck in a loop until they start a new chat.

**The rule:** if a turn announces intent to use a tool, the tool call MUST be in the same `function_calls` block. Never leave intent floating at a turn boundary. If the tool cannot be fired in the current turn (e.g. it depends on a user answer, or it would need to be split across turns), say "I'll do this in my next turn — say go ahead" instead of "Going: <tool>" with no follow-through.

**Diagnostic symptoms the user reports:**
- "I had to start a new chat"
- "No matter what input I gave you, you ignored me"
- "I waited N minutes and got nothing"
- "No pending answer request" appearing in the runtime

**The verbal signal in the agent's own draft:** if the response contains "Going:", "2 tools called", "Dispatching now", "Spinning up", "Will do X next", "Firing tool Y", or any present-tense intent to call a tool AND the function_calls block is empty, that is the anti-pattern. Either add the tool call, or rewrite as "I'll do X in the next turn once you confirm Y" — never both announce and trail off.

**Companion to consent theater (L in hermes-self-improvement):** consent theater is "I need you to click approve" when no dialog exists; announce-without-fire is "I'm firing the tool" when no tool fires. Both are fabrication. The user's pushback in this session: "no matter how much time had passed" — that is the rule, treat it as load-bearing across sessions.

- **The kanban dispatcher validates `--skill` flags against the dispatching profile's catalog, not the target assignee's (NEW 2026-06-27, this user).** When `hermes kanban create` is called from default's gateway, the `--skill <name>` flag is checked against default's skills. If the target profile (e.g. prompt-engineering) has a skill that default doesn't, the spawn dies on first run with `Error: Unknown skill(s): <name>`. The ticket auto-blocks at `failure_limit=2` (default). **Two follow-on errors fire that look like the agent's fault but are config errors:** (a) the worker's log file is empty (the worker died on the validation error before any tool call), (b) the `pid <X> not alive` crash signal in the events table looks like a runtime crash, not a config validation. **Diagnostic sequence:** if a kanban ticket crashes twice in <2 minutes with the same `pid not alive` signal, read the worker's log (`hermes kanban log <id>`) — if the log shows `Error: Unknown skill(s):` instead of a stack trace, it's a config validation failure, not a runtime crash. **Fix:** omit `--skill` flags on `hermes kanban create`; the worker inherits the target profile's skills from its `profiles/<name>/skills/` directory on startup. **`unblock` the ticket and re-create** (or edit the body to remove the skill reference) — don't keep retrying. - **Bash single-quote body content breaks on long `hermes kanban create --body` commands (NEW 2026-06-27, this user).** When the ticket body is 5+ paragraphs with apostrophes, backticks, and `$(...)` patterns, embedding it in a `terminal()` bash command with single-quote escaping fails with `unexpected EOF while looking for matching "'"`. **Fix:** write the body to a Python file in the cwd, then `subprocess.run([...])` from `terminal()` to invoke the CLI. Don't try to embed long bodies in heredoc or single-quote-escaped strings. The Python wrapper pattern is reliable: `cmd = ["cmd", "/c", "hermes", "kanban", "create", title, "--assignee", ..., "--body", BODY]; subprocess.run(cmd, capture_output=True, text=True)`.

- **The dispatcher races the parent's ticket-body updates (NEW 2026-06-27, this user).** Failure shape: parent queues a kanban ticket, then discovers the ticket body has a wrong section (stale job titles, wrong skill references, missing instructions). Parent updates `tasks.body` via SQL `UPDATE tasks SET body = ?`. Meanwhile, the dispatcher's 60s tick has already claimed the ticket, spawned the worker, and the worker is running on the **OLD** body. Parent sees `tasks.status='running'` and thinks the worker has the updated body. It doesn't. **The dispatcher reads `tasks.body` at claim time, not at update time.** Subsequent updates land in the DB but never reach the running worker. **Diagnostic sequence:** (1) read `hermes kanban show <id>` and compare `tasks.body` to your last-known intended body, (2) check `task_runs` for a `spawned` event timestamp — if the spawn timestamp is BEFORE your update timestamp, the worker has the old body, (3) if the worker is still running, you cannot change what it's doing — the only fix is to wait for it to finish (or `hermes kanban block <id>` to kill the run), then queue a correction ticket with the fixed body, (4) if the worker has already finished, queue a correction ticket — the original ticket's `result` is whatever the stale body produced. **The fix pattern is to queue a correction ticket, not to update the original.** Correction tickets should reference the original by `task_links.parent_id`, and the `tasks.body` of the correction should be self-contained (don't say "see ticket X for context" — the worker is spawned fresh and has no view of the parent's history). **Companion to the duplicate-response failure mode:** duplicate-response is "I re-sent the same content twice"; dispatcher-race is "I updated the source but the consumer read the snapshot". Both are stale-context problems; the difference is which layer of the stack holds the staleness.
