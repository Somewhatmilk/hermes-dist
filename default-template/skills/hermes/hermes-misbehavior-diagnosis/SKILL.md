# Hermes Agent Misbehavior Diagnosis


# Diagnose agent misbehavior

The fastest way to lose the user's trust is to assume your own state is correct when it's not. The first move on any "you're doing X wrong" complaint — or any case where you suspect injection, hallucination, or stale context — is to **look at the actual files and processes on this host**, not at Mnemosyne, not at the conversation history.

This skill covers the agent-side of that investigation. The infra-side lives in `session`.

## When to Use This Skill

Use when the user reports **agent misbehavior** that isn't a clean bug:

- Agent claims results that didn't happen
- Agent ignores a clearly-loaded rule
- Agent asks before doing when authorization was given
- Agent hallucinates state (sees files that aren't there, env vars unset, etc.)
- Agent fabricates confidence ("X succeeded" when log shows failure)
- Agent "remembers" the rule after violating it but didn't apply it preemptively
- Agent reports cleanup complete ("removed X", "freed Y GB") without proving the bytes left the host. User pushback phrasing: "is X still there?", "the old Y is still there?", "did u check images too?"
- Agent speculates about a 503 / fallback failure without reading `~/.hermes/logs/agent.log`** (2026-07-09 — see references/503-fallback-chain-diagnostic.md). User pushback: "is it the fallback model messing up my context", "everytime a 503 error occurs i thought u keep context".
- **Agent claims end-to-end merge success without per-file compatibility check (NEW 2026-07-10).** When the user asks "did u validate my custom code against the new hermes BEFORE pushing to local", the answer must be a per-file DROP/KEEP/REBASE walkthrough with grep evidence, not "hermes doctor passed" or "all checks pass" or "Up to date." User pushback verbatim: *"did u verify and validate inside the sandbox working with the addition and the new hermes update code then push it to my local? or did all u do was in the sandbox call git pull on hermes-agent there is a difference, what i want is validation on my custom code compatibility with the new hermes before pushing to local"*. See `references/anti-pattern-merge-success-without-per-file-check.md`.
- **Agent fires `clarify` after real-time delegation (NEW 2026-07-09).** When the user says "do what u see fit" / "whichever is better long term" / "either X or Y, just do it" within the same session, the agent must pick and execute — not re-ask. Distinct from the existing "ask for already-given permission" pitfall in `references/clarify-loop-resumption.md` (which is about permission carried over from a PRIOR session); this is about delegation given in real time within the current session. User pushback phrasing: "dont kill them just change the setting and tell me your done that it", "do what u see fit and tell me when done", "either removee the override or do what u see fit and hcange whats inside whichever is better long temr".
- Agent re-quotes a rule the user already said to stop quoting (NEW 2026-07-09).** If the user said "stay off with that quote from now on" or any variant of "I've heard this already — stop", the rule itself is preserved in memory and the relevant skill; just enforce the behavior silently, no preamble. See `references/self-tics-2026-07-09.md`.
- **Agent validates a phantom complaint (NEW 2026-07-11).** When the user (or a delegated subagent, or a parent prompt) delivers a complaint that names a specific file and proposes a patch, the agent must `ls` / `find` / `search_files` that file BEFORE writing any analysis. The most common failure mode is analysing a fix for a file that does not exist on disk — the analysis may be correct in general, but the patch is untargetable. User pushback: *u should validate the complaints first before recommending fixes*, *what's the actual file path, not the imagined one*, *the file doesn't exist*, *is this complaint about a real file or did you make it up?*. Self-tic: *the patch is technically correct in general*, *applying this to the file would address...*, *the script at <path> line <N> needs...* (when <path> was never confirmed to exist). See `references/anti-pattern-phantom-complaint-validation.md`. Companion to predicting-a-cause-without-running-the-diagnostic (v2.7.0) — same handle-demanding discipline, different failure mode.
- **Agent reaches for `%TEMP%` or `AppData/Local/Temp/` instead of `~/Downloads/<topic>_<YYYY-MM-DD>/` (NEW 2026-07-09).** The 0.85 working-artifact rule says scratch goes to Downloads/. The 2026-07-08 AND 2026-07-09 sessions both made this mistake. See `references/self-tics-2026-07-09.md` and the umbrella `hermes-config-cli-gotchas/references/working-artifact-lifecycle-2026-07-09.md`.

## When NOT to Use

- Pure tool errors (use `failures-journal`)
- Performance issues (use `hermes-misbehavior-diagnosis` only if the
  slowness is **caused by misbehavior**, not infrastructure)
- Bug in user code (not the agent's job)

## Philosophy (load-bearing)

1. **Diagnosis is the work, not the answer.** Most misbehavior is
   upstream of the agent's response — tool env, memory state, prompt
   mutation, scheduler interference. The agent's draft reply is the
   symptom, not the cause.
2. **Verify by reading the code, not by reading the parser.** When a
   CLI verb misbehaves, `cmd_<verb>` in `main.py` is behavior;
   `subcommands/<verb>.py` is argparse plumbing. Confusing the two is
   the most common misdiagnosis.
3. **Memory recall ≠ rule enforcement.** A rule that lives only in
   working memory is advisory. Operationalize as precondition check
   at dispatch time. (See `references/memory-recall-rule.md`.) For the
   per-project-recall failure mode specifically (cross-project
   contamination when the user runs multiple parallel agent projects),
   the operational fix is the `per-project-context` skill at
   `~/.hermes/skills/devops/per-project-context/SKILL.md` — see its
   Changelog entry 2026-07-02 for the user's full feedback that
   3. **Memory recall ≠ rule enforcement.** A rule that lives only in working memory is advisory. Operationalize as precondition check at dispatch time. (See `references/memory-recall-rule.md`.)    4. **"I cleaned up X" is a claim, not a fact, until `df`/`docker system df`/filesystem re-prove it.** When a session removes a container, file, or image, the next session must re-verify with the same probe used in the inventory. The 2026-07-03 incident (claimed playwright-research cleanup complete; user pushed back "the older container playwright research i still there?") showed the gap: container removal ≠ image removal ≠ volume removal. The cleanup ritual must include `docker ps -a` + `docker images -a` + `docker volume ls` + `docker system df` BEFORE claiming any "freed X GB" number to the user. Full pattern in `references/anti-pattern-cleanup-claim-without-reclaim-verification.md`.
   5. **"I think X is happening" about a 503 / fallback failure is a claim, not a fact, until you've grepped `~/.hermes/logs/agent.log` for `503`, `Fallback activated`, and `<PROVIDER>_API_KEY exhausted`.** When the user reports a "continue" prompt arriving with no prior context, the temptation is to speculate about the fallback model. The 2026-07-09 incident showed this is almost always wrong: the fallback model never loads, the gateway injects "continue" as a recovery placeholder after both primary AND fallback fail. The fix is a 3-grep diagnostic, not a 3-paragraph speculation. Full pattern in `references/503-fallback-chain-diagnostic.md`.
