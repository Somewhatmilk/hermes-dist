# Batch Secret Migration to `pass` — Values-via-Stdin Recipe

When migrating N secrets from `~/.hermes/.env` (or anywhere) into
`~/.password-store/`, you want a workflow that:

1. **Never puts a secret value on any command line** (avoiding bash
   history + `ps`/`tasklist` leakage).
2. **Survives sandbox / TTY differences** between `terminal` tool
   (interactive shell, has PATH) and `execute_code` (sandboxed, no PATH).
3. **Gives a clear pass/fail per entry** so you can spot stuck entries
   without re-running the whole batch.
4. **Verifies by length, not by value** — never echo a decrypted secret
   back to chat.

## Prerequisites (already covered in install recipe)

1. `pass` installed at `~/bin/pass` (single-file copy from
   git.zx2c4.com).
2. GPG keypair generated + `pass init <email>` ran.
3. **`~/.gnupg/gpg-agent.conf` has `allow-loopback-pinentry` + ttl=0**
   (otherwise all inserts fail silently with rc=1).
4. `gpg-agent` restarted (`gpgconf --kill gpg-agent`).
5. **gpg-agent primed with YOUR passphrase** — see "Step 0" below.
   Without this priming, the loader's decrypt verification round-trip
   fails with `gpg: Sorry, we are in batchmode - can't get input`.

> **2026-07-05 CORRECTION** (replaces the original Step 2 loader):
> `pass insert` ALWAYS opens an interactive TTY prompt via
> `read -s` in its bash source (around line 463 of the script). It
> does NOT respect piped stdin — even with `allow-loopback-pinentry`
> enabled. Therefore this recipe calls `gpg --encrypt` directly
> and writes the `.gpg` file yourself, bypassing `pass insert`.
> This is functionally identical (it's what `pass insert` does
> internally) but works in non-TTY contexts. Reserve `pass` for
> read/ls operations only; never `pass insert` from a script.

## Step 0 (one-time per session) — Prime gpg-agent with your passphrase

`gpg --decrypt` needs the private key unlocked. When run from a
script without an interactive TTY (sandbox, piped heredoc, agent
`terminal` tool when Hermes is running headless), the agent CAN'T
display the pinentry popup. So YOU prime the agent yourself once:

1. Open your own terminal (git-bash / PowerShell) and run **one**
   command that touches the private key:
   ```bash
   gpg --clearsign < /dev/null
   ```
   (`gpg --list-secret-keys` does NOT count — read-only.)
2. Type your passphrase in the Kleopatra / pinentry popup.
3. gpg-agent caches the passphrase for the session
   (`default-cache-ttl 0` keeps it cached until logout).

Without this priming step the migration script's per-secret
verification decrypt fails silently. Files get encrypted but every
length check returns `0/N`.

## Step 1 — write the values file (0600, in `~/bin/` or elsewhere)

Pipe-delimited, one entry per line: `<pass-path>|<value>`.

```bash
cat > ~/.bin/.pass-secrets.tmp <<'EOF'
api/openrouter|sk-or-v1-…
api/spotify-client-id|29e9e72…
api/telegram-bot|8720…
api/discord-bot|MTUx…
api/minimax|sk-cp-…
api/mnemosyne-llm|ollama
api/server-key|3614…
api/civitai|f52f…
EOF
chmod 600 ~/.bin/.pass-secrets.tmp
```

> Generated by the agent via the `write_file` tool (doesn't echo to
> terminal). The 0600 mode means only `somew` can read it.

> **Trim values from agent chat history** if the user pasted in chat: use
> the Mnemosyne redaction format `<{NAME}:{len8}chars:{first6}…{last4}>`
> for any displayed record.

## Step 2 — write the loader script (0700)

A small bash script that reads stdin one line at a time, splits on
`|`, encrypts each value with `gpg` directly (NOT `pass insert`,
which requires an interactive TTY), and verifies by length.

```bash
cat > ~/.bin/.pass-loader.sh <<'SCRIPT'
#!/bin/bash
set +o history
unset HISTFILE
export PASSWORD_STORE_DIR="$HOME/.password-store"

# Get recipient fingerprint from pass init's .gpg-id
FPR=$(cat "$PASSWORD_STORE_DIR/.gpg-id" 2>/dev/null | head -1)
if [ -z "$FPR" ]; then
  echo "ERROR: no .gpg-id in $PASSWORD_STORE_DIR"
  exit 1
fi

mkdir -p "$PASSWORD_STORE_DIR"

while IFS='|' read -r path val; do
  [ -z "$path" ] && continue
  [[ "$path" =~ ^# ]] && continue
  outfile="$PASSWORD_STORE_DIR/$path.gpg"
  expected=$(printf '%s' "$val" | wc -c | tr -d ' ')

  # Encrypt directly with gpg, output to .gpg file. This is what
  # `pass insert` does internally but works without a TTY.
  if printf '%s' "$val" | gpg --batch --yes --quiet \
        --pinentry-mode loopback \
        --trust-model always \
        --encrypt --recipient "$FPR" \
        --output "$outfile" 2>/dev/null; then
    # Verify by decrypting to a temp file, check length, delete.
    # CRITICAL: redirect stdout to a file. `$(gpg ...)` would capture
    # BOTH the metadata warning lines ("gpg: encrypted with...") AND
    # the plaintext, contaminating length check even with --quiet.
    # Also: do NOT pass --batch on decrypt — it's mutually exclusive
    # with --pinentry-mode loopback (see gpg-gotchas.md). After Step 0,
    # gpg-agent has the passphrase cached, so plain `gpg --decrypt`
    # works without either flag.
    tmpfile=$(mktemp)
    if gpg --yes --quiet --decrypt "$outfile" > "$tmpfile" 2>/dev/null; then
      actual=$(wc -c < "$tmpfile" | tr -d ' ')
      if [ "$actual" = "$expected" ]; then
        printf 'OK    %-28s %s/%s bytes\n' "$path" "$actual" "$expected"
      else
        printf 'FAIL  %-28s %s/%s length mismatch\n' "$path" "$actual" "$expected"
        rm -f "$outfile"
      fi
    else
      printf 'FAIL  %-28s decrypt verification error (gpg-agent primed?)\n' "$path"
      rm -f "$outfile"
    fi
    # Best-effort secure delete: overwrite with random, then unlink.
    dd if=/dev/urandom of="$tmpfile" bs=1 count="$(wc -c < "$tmpfile" | tr -d ' ')" 2>/dev/null
    rm -f "$tmpfile"
  else
    printf 'FAIL  %-28s gpg encryption error\n' "$path"
  fi
  unset val
done
SCRIPT
chmod 700 ~/.bin/.pass-loader.sh
```

**Three things to notice:**

1. `set +o history` + `unset HISTFILE` — keeps the loader out of
   shell history even if invoked from an interactive shell.
2. The fingerprint comes from `$PASSWORD_STORE_DIR/.gpg-id` which
   `pass init` wrote — single source of truth for the recipient.
3. The verification decrypt writes to a **tempfile**, not captured
   by `$(...)`. Reason: `gpg` emits metadata warnings like
   `gpg: encrypted with rsa4096 key, ID ..., created ...` to stderr
   by default, but when stderr is captured together with stdout the
   shell sees warnings as plaintext. `--quiet` alone does NOT suppress
   these metadata lines. Decrypt → tempfile → `wc -c < tempfile`
   avoids the contamination entirely.

## Step 3 — run it

From the `terminal` tool (PATH + TTY intact — gpg-agent primed):

```bash
cat ~/.bin/.pass-secrets.tmp | ~/.bin/.pass-loader.sh
```

Output: one line per entry:
```
OK    api/openrouter                73/73 bytes
OK    api/spotify-client-id         32/32 bytes
OK    api/telegram-bot              46/46 bytes
...
```

**Failure modes and their meanings:**

| Symptom | Cause | Fix |
|---|---|---|
| All FAIL with `gpg encryption error` | `gpg-agent` not running or fingerprint wrong | `gpgconf --kill gpg-agent && gpg-agent --daemon`; check `~/.password-store/.gpg-id` |
| All FAIL with `decrypt verification error` | gpg-agent not primed with passphrase (Step 0 skipped) | Run `gpg --clearsign < /dev/null` in your own terminal to trigger pinentry |
| All files written but FAIL with `0/N length mismatch` | Decrypt returned 0 bytes — `--batch` + `--pinentry-mode loopback` conflict on decrypt (gpg says "we are in batchmode - can't get input") | Drop `--batch` and `--pinentry-mode loopback` from the decrypt line, leave only `gpg --yes --quiet --decrypt` |
| One entry FAIL with length mismatch | Output pipe or line truncation in `.pass-secrets.tmp` | Check for stray `|` or newlines; rerun on just that entry |
| Length-mismatch shows value bigger than expected (e.g. +48 bytes) | Stderr metadata "gpg: encrypted with..." leaked into stdout capture | Decrypt to tempfile, never `$(gpg ...)` for length checks |
| `pass show` hangs / times out | gpg-agent passphrase cache dropped (timeout, agent idle) | Re-prime via `gpg --clearsign < /dev/null` |

## Step 4 — verify (length only, NEVER show values)

```bash
pass ls
# output:
# Password Store
# └── api
#     ├── openrouter
#     ├── spotify-client-id
#     ...

pass show api/openrouter | wc -c    # should match the original length
```

NEVER pipe `pass show` output back to chat. The agent's job is to
*use* the secret (pipe it to a consumer, not look at it).

## Step 5 — delete the temp files

```bash
shred -u ~/.bin/.pass-secrets.tmp   # best-effort secure delete
shred -u ~/.bin/.pass-loader.sh
```

Or use `~/.hermes/bin/soft-delete.sh <path>` per the project's
soft-delete discipline (recoverable via `~/.hermes/trash/<date>/`).

## Step 6 — redact `~/.hermes/.env`

Replace each `KEY=value` line with `KEY=` (empty value), preserving
the variable name so any legacy `os.environ.get("KEY")` callers don't
KeyError. Variable names themselves can stay; the secrets are gone.

`~/.hermes/.env` is **write-blocked** for the agent. Surface the
patch to the user with the existing vars + their redacted replacements
and let them paste it in.

## Why this recipe and not other approaches

| Approach | Why we don't use it |
|---|---|
| `pass insert <path>` (interactive TTY) | Hardcoded `read -s` prompt — fails silently on any non-TTY context (scripted, piped, sandboxed). Do NOT use from scripts. |
| `pass insert <path> <value>` on argv | Hits bash history + `ps`; surfaced in `~/.bash_history`. Multi-line values break argv. |
| `gpg --encrypt` via subprocess from `execute_code` | Same argv leak. Use `subprocess.run([..., path], input=value.encode())` if doing it from Python, but the bash loader script is shorter. |
| `gpg-preset-passphrase` + `--passphrase <val>` | Passphrase leak surface; complex setup; no benefit if you prime gpg-agent once at session start (Step 0). |
| `expect` script | Adds a dependency, hard to audit, doesn't survive Windows console quirks cleanly. |
| `git credential-store` / `keyring` / `Secret Service` (libsecret) | Adds 1–2 dependencies; not portable to Windows without MSYS hacks. |

## The pattern in one line

> **Values flow through stdin to a 0700 loader script that calls
> `gpg --encrypt` directly (NOT `pass insert`), writing the
> `.gpg` file to `~/.password-store/`. They never appear on any
> command line.**
